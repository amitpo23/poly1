## Activating scalper Stage 1 (operator-only)

The scalper has been in shadow at the new `$2.50` leg size since
2026-05-06. Per `SPEC.md §15`:

> Stage 1 (live small): EXECUTE_SCALPER=true, leg=$2.50,
> **min 2 weeks**. Abort if cumulative PnL < -$15 at any point.

### When to flip Stage 1

After **at least 24 hours** of `$2.50` shadow that produced a
non-trivial trigger count. Specifically:

- ≥ 3 SHADOW rows in `trades` table from yesterday → enough activity.
- Trigger reasons distributed across `depth` and `reversal` (not all
  one type) → strategy is firing on both pathways.

If shadow is silent (0-1 triggers in 24h), **do not flip** — the
strategy is starved and live mode would only confirm that.

### Pre-flip check

The scalper is in a separate container (`profile: scalper`). Capital
isolation is via `SCALPER_RESERVE_USDC` in `.env`.

```bash
grep -E "^SCALPER_RESERVE|^SCALP_LEG_USDC|^EXECUTE_SCALPER|^MAX_SCALP" \
  ~/coding/poly1/.env
```

Expected before flip:
- `SCALPER_RESERVE_USDC` set to ≥ leg × 4 (so the scalper has enough
  cash for at least 2 full pairs without competing with poly1 main).
- `SCALP_LEG_USDC=2.5`
- `MAX_SCALP_TRADES_PER_HOUR` set to a reasonable cap (default 60).
- `EXECUTE_SCALPER=false` (we'll flip this).

### Flip


### First-week criteria

Stage 1 lasts a **minimum of 2 weeks**. Within those 2 weeks:

- Cumulative scalper PnL < `-$15` at any point → **abort Stage 1**.
- Any `RECONCILE_NEEDED` scalper_pairs row → **stop, verify on-chain
  positions, clear manually**. The scalper expanded its on-chain
  exposure and we no longer trust the local state.
- After 2 clean weeks → consider Stage 2 (leg=$5+).

## Halt rules

### Per-bot

- poly1: `touch ~/coding/poly1/data/HALT` — RiskGate reads this and
  blocks all entries.
- scalper: same HALT file (shared trade_log + RiskGate).
- swarm: SIGTERM the container; coordinator drains gracefully.

### Cross-bot

If **both** poly1 and swarm show drawdown > 5% same day → halt both.
If only one — the halted bot's RiskGate handles it; the other keeps
running on its own envelope.

This is a manual policy, not automated. Each bot's RiskGate only sees
its own DB.
